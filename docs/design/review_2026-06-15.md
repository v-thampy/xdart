## 1. Batch vs. live — and the correction that matters for notebooks

There are **three architecturally distinct reduction paths**. The perf-baseline "Batch" row predicts
none of the notebook ones.

| Path | How it reads frames | Overlap / parallelism |
|------|--------------------|----------------------|
| **xdart Live** (`flush_size=1`) | 1 background prefetch thread, queue depth 4; read→reduce interleaved per frame | reads hide behind reduce ✅ |
| **xdart Batch** (`flush_size=64` 2D / `256` 1D) | same prefetcher, but **collect-then-dispatch barrier per chunk** | prefetcher (4 deep) can't bridge a 64-frame reduce phase → reads **exposed** ❌ |
| **Notebook `run_reduction(...)` DEFAULT** | `execution="chunked", chunk_size=1, executor=None` → fully **serial** | no overlap, no parallelism ❌❌ |
| **Notebook streaming** (`execution="streaming", executor=True`) | workers call `frame.load_image()` themselves | parallel decode across workers, fully overlapped ✅✅ |

### Why xdart Batch is ~50% slower than Live
The slowdown is **not in the reduction** — it is that batch stops overlapping disk reads with reduce.
The perf table proves it: for 2D, `dispatch` (the pyFAI cake) is identical (Live 34.1 vs Batch
35.1 ms/fr), but `read` jumps from **3.3 ms/fr hidden (Live) → 19.2 ms/fr exposed (Batch)**.

Root cause, three interacting facts in `image_wrangler_thread.py`:
1. Batch accumulates `flush_size` frames into `pending` **before** dispatching the whole batch
   (`_PENDING_FLUSH_SIZE = 64`, `_PENDING_FLUSH_SIZE_1D = 256`, lines 292/297, gate at 821-844). Live
   uses `flush_size = 1` and dispatches every frame immediately.
2. The Eiger prefetch queue holds **individual frames** with `XDART_PREFETCH_QUEUE_SIZE = 4`
   (line 318; frames pushed one at a time at line 2289), so it can only run ~4 frames ahead.
3. Result: during a 64-frame read phase nothing is reducing (prefetcher fills 4 slots and stalls);
   during the reduce phase only 4 of the next 64 reads are hidden. Reads and reduce run in phases.
   The code comment at lines 306-312 already hypothesizes exactly this; the table confirms it.

A second, deeper asymmetry: xdart reads **single-threaded** (one prefetch thread doing bulk HDF5
slices) and hands the already-decoded array to the workers (`LiveFrame(img_number, img_data, …)` in
`_build_batch_frames`, line 1741), so the worker pool never decodes — it cannot break the
"single-thread decode ceiling ~16 s" the perf doc notes.

### The notebook correction
The **headless default is the slowest path of all.** A naive `run_reduction(plan, scan, NexusSink(...))`
gets serial `chunk_size=1` (`reduction/core.py:1467-1481`), which also **defeats Eiger bulk
decompression** in `NexusStackSource.iter_chunks`. The fast streaming path exists but is opt-in. Since
notebooks are the headless story, this is arguably the **single most impactful perf item** even though
the reviewers scored it P2.

### Recommendations (after 7+8)
1. **Flip `run_reduction`'s default to `execution="streaming"` when a durable (non-Memory) sink is
   supplied** — there is precedent for auto-selecting on that exact condition (`retain_products`), so
   it is low-risk; `_init_streaming` self-builds an owned pool. At minimum, document loudly that the
   default is serial. *(S — biggest notebook win)*
2. **xdart batch: submit-per-read.** Decouple the submit cadence from the disk-save flush cadence —
   build the `LiveFrame` and `adapter.submit()` the instant each frame is read; keep the 64/256
   interval only for `FlushPolicy` disk saves. The persistent session's in-flight semaphore already
   provides backpressure, so the `pending` chunk is redundant. Restores live-style overlap; folds into
   **Phase 4e**. *(M — the real fix)*
3. Confirmation experiment: raise `XDART_PREFETCH_QUEUE_SIZE` and/or `inflight_max` (default
   `2×workers` ≈ 16) for batch and watch `collect_read` collapse. Not the fix (costs ~1 GB buffered
   raw), but locks the diagnosis. *(S)*
4. **Eiger prefetch widens to `int32` unconditionally** (line 2253) — read native dtype; halves queue
   RAM for uint16 detectors and drops one full-frame copy/frame. *(S)*

The GI common-grid freeze does not block submit-per-read: it already scout_unions first+last frames in
`_gi_freeze_whole_scan_prepass` (lines 1739/1783), independent of submit cadence.

## 2. Steps 7+8 readiness — the urgent part (the live work)

The design is sound and the reviewers agree with its direction. These are the gates 7+8 must clear,
ordered.

**A. No chunked aggregation reader yet — and `get_*(frame=None)` is a 2× full-stack alloc.**
`_slice_stack` (`io/read.py:608`) reads the whole stack, then does an **unconditional reorder copy**
that is an identity no-op when `frame=None` (positions already sorted) → ~1.9 GB transient for a
651-frame 2D cake, multi-GB at scale. **7b must not call `get_2d(frame=None)`.** Build a streaming
accumulator that folds `nansum`/`nancount` over position chunks (32–64 frames) — the fancy-index API
already supports `get_2d(file, frame=<chunk of labels>)`. Also fix the no-op reorder copy. *(P1; M)*

**B. 7b must apply per-frame normalization before reducing.** If a `normChannel` is selected, a naive
disk-stacked `nansum`/`nanmean` is **silently wrong** — it skips the per-frame divide
`get_frames_int_1d` does today (`display_data.py:568-643, 862-881`). Read aligned `scan_data`,
normalize each row, *then* reduce. **Add a normChannel-on aggregation parity test.** *(P1 correctness; M)*

**C. The 2D "Overall" cake still averages only the resident subset.** The 1D payload was hardened to
check store residency against the full intended set (`display_publication.py` integration_plot_payload),
but the **2D cake path was not** — it still uses `render_ids` (cake_image, ~lines 306-339), so an
evicted frame silently drops from the Overall cake. Mirror the 1D guard (blank, do not average-wrong)
until on-disk aggregation lands. Do **not** relax the guard. *(P1; M)*

**D. Ordering hazard — do NOT land 8b (delete `data_1d`) before 7b is verified.**
`data_1d = FixSizeOrderedDict(max=0)` (`static_scan_widget.py:329`) is the *live unbounded aggregation
backstop* that currently hides the Round-12 >64-frame truncation bug. Gate 8b on a checklist, not just
spine-green: (1) >64-frame aggregation test green, (2) normChannel parity green, (3) 2D chunked-read
peak-RAM bounded, (4) live-read vs `file_lock` race resolved. This matches the design's own
"live checkpoint required for 8a/8b/7c." *(P1)*

**E. Test gaps to fill *with* 7b, not after:**
- **No >64-frame whole-scan Sum/Average test** exists (coverage is N=30–40); this is the literal
  Round-12 regression test — land it first, modeling the production flow (evict past caps, aggregate
  over ALL frames, assert == analytic reference, assert off-GUI-thread). Parametrize 1D + 2D. *(P1)*
- The **equivalence spine is single-frame only** (`test_gi_batch_real_data.py` reduces `_TIFF_FRAMES[0]`
  only) — add a multi-frame stack + multi-mode aggregation variant (the ADR-0005 gate). *(P2)*
- **Off-GUI-thread aggregation no-freeze** and **2D persist-before-evict under the new cadence** are
  untested (all persist-before-evict coverage is 1D, `skip_2d=True`, N≤60). *(P2)*
- **No test enforces the primary-mode-scoped guard** — a non-primary GI whole-scan aggregate must
  *refuse* a partial stack (re-introduces Round-12 if unguarded). *(S)*

**F. Net-new work the design names but isn't built:**
- 7a (the headless `FrameRecord` store) does not exist yet — the session holds *no* record store
  (`session/scan_session.py:174-225`, `retain_products=False`). 7a is net-new, not a refactor.
- The "W" multi-mode write-wiring is unbuilt → non-primary GI modes are in-memory-only and lost on
  reload. And `get_1d/get_2d` have **no `mode=` parameter**, so once "W" writes per-mode subgroups the
  stacked reader can't see them — add the parameter alongside W.
- The session docstring still says cadence is an xdart concern (`scan_session.py:19-22`) — 7c/ADR-0005
  reverses that; update it.
- The xdart `ScanSession` 4f-bridge is wired only into `imageWranglerThread`; `nexusThread` still drives
  a raw chunked `ReductionSession` with its own write/save loop (`nexus_wrangler_thread.py:287-345`).

## 3. Memory pressure (48 findings) — highest-leverage wins

### Hot-path allocation churn (fix regardless of 7+8)
- **`bai_1d`/`bai_2d` running-sum accumulators are dead weight** — a full 2D-slab `__add__` *every
  frame* (~10 GB churn on a 651-frame 2D scan), and the only consumer is `test_ewald.py` (no
  production reader; the sibling `overall_raw` accumulator was already removed for this reason, G2
  comment). Delete `_accumulate_bai_*` + the attrs and update the test. `ewald/scan.py:443-486`.
  *(P1; **S — easy win**)*
- **Per-frame raw cast to float64** in `_reduce_frame` (`core.py:2056`) and **2D cake always
  materialized float64** before the writer downcasts to float32 (`integrate/single.py:181`,
  `multi.py:212`) — doubles the working set with no precision benefit for pyFAI. *(P2; M)*
- **Background subtraction allocates a full float64 frame even when bg is scalar 0** (the default,
  `core.py:2165-2176`); **thumbnail recomputes `(raw - bg)`** as two fresh full-frame float32 arrays,
  redundant with the subtraction core already did (`ewald/frame.py:893`). *(P2; S/M)*

### Whole-stack-in-RAM loads (OOM at scale — headless too)
- **`integrate/batch.py::process_scan` reads the entire stack up front + float64-casts it**
  (`batch.py:118-124`); **`io/image.py::read_images_parallel` / `read_image_stack(reduce=...)`
  list-then-stack double** the whole stack (lines 227-303) — and `read_image_stack` cannot serve as the
  streaming aggregation primitive 7+8 wants. Make these stream one frame at a time. *(P1/P2; M)*
- **Batch phase-fit retains a full lmfit `ModelResult` + fitter per frame** — unbounded over a scan
  (`analysis/fitting/batch.py:172,344`); `fit_nexus` also materializes the full (N, n_q) stack. Add a
  lightweight-summary mode and default the whole-scan paths to it. *(P1; M)*
- **RSM `combine_grids` materializes ~0.5 GB of transient meshgrid** for default 200³ bins
  (`rsm/gridding.py:595-606`). *(P2; M)*
- **Persistent streaming session accumulates every `Frame` (with metadata) in
  `scan.frames`/`_frame_by_index` for the whole scan, never released** (`core.py:1304-1345`). Bounded
  but grows to N; relevant at 10k frames. *(P2; M)*

### Unbounded caches (the 7+8 targets — confirms D2/D3)
- `data_1d` unbounded with a **float32 262 KB thumbnail per entry** (should be uint8 64 KB to match
  the persisted dtype); the full-reload `_absorb_chunk` path materializes all N at once
  (`static_scan_widget.py:329`, `h5viewer.py:2416`). Quantizing the thumbnail is a cheap interim before
  the store migration completes. *(P1)*
- **PublicationStore `_items` unbounded (`max_items=None`)** with redundant per-frame metadata copies
  (~4–5 dicts/frame, multiplied per GI mode — D3) (`frame_publication.py:158-164, 481-504, 574-589`).
  As 8a makes the store a derived projection, share one `MappingProxyType` for raw + one for numeric
  and reference `view.metadata_*` instead of re-copying. *(P1; M)*

### Reintegrate-All (D1) — confirmed unfixed but dormant (buttons hidden)
Persist-before-evict pins all N reintegrated frames; the single end-of-run replace-save is a second
whole-scan peak (`scan_threads.py:347-376,450-459`; `ewald/nexus_writer.py:573-591`;
`ewald/frame_series.py:494-508`). New angle: the "batched dispatch" does *not* bound RAM. Fix
(prerequisite to re-enabling the buttons): chunked replace-save + `mark_persisted(chunk)` per batch +
null `live_frame.map_raw/bg_raw/mask` right after `_publish`. *(P1; L)*

### Other retention
- **`stash()` silently skips ALL eviction when nothing is persisted yet** — the cache can exceed
  `_in_memory_cap` unbounded if a dispatcher ever stops forcing saves (`frame_series.py:494-508`). *(P2; M)*
- **Image-viewer browse pins full ~18 MB raw arrays in `data_1d`** that the hydrated-raw LRU never
  frees (`h5viewer.py:1916-1987`). *(P2; M)*

## 5. Correctness & concurrency

- **`integrate/batch.py::process_scan` forwards one `**kwargs` blindly to both `integrate_1d` and
  `integrate_2d`** — any 1D-only or 2D-only pyFAI kwarg aborts every frame (`batch.py:234-260`). *(P2; S)*
- **RSM corner-only scout can under-bound the q grid** → out-of-range bins silently dropped
  (`rsm/gridding.py:128`). **March-Dollase texture on a structure-less (fixed-q) phase uses a
  placeholder unit-cubic metric tensor** → wrong corrections (`phase_fitting.py:625-629`). *(P2; S)*
- **Reintegration write failure on `finish()` is recorded to a dead flag and never surfaced** to the
  user (`scan_threads.py:120-133`). *(P2; S)*
- **`H5FilePool.pause/resume` is not refcounted** (plain set, `utils/h5pool.py:69-87`) — a
  nested/concurrent pause resumes early and can reopen the file mid-write. Make it a `Counter`; cheap,
  backward-compatible, and important before the 7+8 flush fan-out makes overlaps routine. *(P1; S)*
- **Streaming `_failure`/`_cancelled` flags written by the writer thread are read by `submit()` without
  a barrier/lock** (`core.py`). Low-probability on CPython; worth a lock. *(P2; S)*
- **Live-run stacked reads will race the xdart writer's `file_lock`** — headless `get_1d/get_2d` use
  plain `h5py.File('r')` with no SWMR and no lock (`io/read.py:251,286`). Resolve before 7b reads a
  live, in-progress file off-thread. *(P2; M)*
